# db_report Skill 评估方案提交说明

## 一、提交目标

本目录用于存放 `db_report` Skill 的评估方案提交材料。提交内容围绕该 Skill 在数据库性能测试报告生成场景下的行为质量进行设计，覆盖理想态定义、评分规则、测试用例设计和元测试设计四部分。

## 二、文件清单

本目录当前包含以下文件：

- `db_report_logic_chain.md`
- `db_report_ideal_state.md`
- `db_report_rubrics.yaml`
- `db_report_testcases.yaml`
- `db_report_meta_testcases.md`

## 三、文件说明

### 1. `db_report_logic_chain.md`

用于梳理 `db_report` Skill 的执行链路，包括目标、输入模式、阶段输出、分流逻辑、门控设计以及停止条件。该文档作为测试用例设计和理想态定义的上游依据。

### 2. `db_report_ideal_state.md`

用于说明 `db_report` Skill 的目标、输入输出、支持的数据源、支持的报告类型、完整执行链路、禁止行为以及异常处理要求，并给出满分 Skill 应达到的理想状态描述。

### 3. `db_report_rubrics.yaml`

用于定义 `db_report` Skill 的结构化评分规则。内容包括评分维度、权重、满分标准、扣分点和严重违规项，总分为 100 分。

### 4. `db_report_testcases.yaml`

用于定义面向 `db_report` Skill 的测试用例集。内容覆盖正常路径、失败路径、边界条件和红线场景，是本评估方案的核心交付物之一。

### 5. `db_report_meta_testcases.md`

用于定义测试集自检方案，即“测试用例的测试用例”。内容主要用于检查测试集是否覆盖充分、判定是否清晰、结构是否完整、是否存在重复或不可判定用例。

## 四、测试集概览

本评估方案当前为 `db_report` Skill 设计了 **12 条结构化测试用例**。测试集以“**用尽量少的用例覆盖尽量多的关键能力与风险场景**”为设计原则，不追求堆叠数量，而强调以下三点：

- 覆盖 `db_report` 的完整主链路：输入解析、数据接入、数据质量门控、按报告类型分流分析、图表生成、三格式交付。
- 同时覆盖正常路径、失败路径、边界条件和红线违规，避免测试集只验证“会不会做”，而忽略“何时必须停、何时必须拒绝”。
- 让每条用例都具备明确的评测目标和可执行 judge，减少重复用例，提升单条用例的覆盖效率与自动化判定价值。

### 1. 测试集结构分布

当前 12 条用例按覆盖目标可分为四组：

- **主路径能力验证（4 条）**：覆盖 `single`、`comparison`、`iteration`、`custom` 四类报告的端到端生成能力。
- **失败路径与红线验证（4 条）**：覆盖缺少数据源、文件不存在、关键指标空值、禁止通过内部标识补查数据等必须停止或拒绝的场景。
- **意图解析与筛选逻辑验证（2 条）**：覆盖 `OR` 条件和 `AND` 条件解析，验证 `intent.json`、本地筛选逻辑和结果集范围是否正确。
- **交付质量与洞察治理验证（2 条）**：覆盖三格式交付一致性、最小交付标准以及 `L1/L2/L3` 洞察分级与位置约束。

### 2. 用例编号与覆盖目标

各测试用例的定位如下：

- `DB-01`：本地 `log` 输入的 `single` 单次测试报告主路径。
- `DB-02`：本地 `records JSON` 输入的 `comparison` 多产品对比报告主路径。
- `DB-03`：本地历史 records JSON 输入的 `iteration` 演进趋势报告主路径。
- `DB-04`：围绕用户专项问题展开的 `custom` 定向分析报告主路径。
- `DB-05`：缺少本地文件和粘贴 JSON 时，必须停在门控①并要求补充数据。
- `DB-06`：用户提供本地路径但文件不存在时，必须在阶段 2 前置校验停止。
- `DB-07`：关键指标空值触发门控②失败，不得继续生成正式分析和报告。
- `DB-08`：仅提供 `task_id / report_id / plan_id` 时，必须拒绝内部补查数据。
- `DB-09`：`OR` 条件解析与多场景筛选逻辑验证。
- `DB-10`：`AND` 条件解析与交集筛选逻辑验证。
- `DB-11`：`md/docx/html` 三格式交付一致性与最小交付标准关键项验证。
- `DB-12`：`L1/L2/L3` 洞察分级、来源可追溯性和位置边界验证。

### 3. 与 Skill 关键能力的对应关系

从能力视角看，当前测试集已经覆盖 `db_report` 的核心要求：

- **数据源覆盖**：覆盖本地 `log`、本地 `JSON`、粘贴 `JSON` 失败场景，以及缺少数据源、错误路径等输入异常。
- **报告类型覆盖**：覆盖 `single`、`comparison`、`iteration`、`custom` 四类报告类型。
- **阶段门控覆盖**：覆盖门控①（缺数据阻断、AND/OR 意图解析）、门控②（关键指标空值阻断）、门控③（三格式交付与最小交付检查）。
- **指标处理覆盖**：覆盖 `TPS / QPS / P95 / P99` 的抽取、追溯、一致性和空值拦截。
- **红线覆盖**：覆盖禁止内部数据库/API 查询、禁止通过 `task_id / plan_id / report_id` 补查、禁止编造性能指标、禁止在数据不足时继续出正式报告。
- **交付质量覆盖**：覆盖图表数量基线、三格式输出、一致性校验、占位符清理和洞察位置治理。

### 4. 设计取舍说明

本测试集没有进一步拆分为大量细碎 case，而是优先采用“**一条用例覆盖一条完整风险链**”的设计思路。例如：

- `DB-01` 到 `DB-04` 不仅验证报告类型识别，还同时验证数据接入、质量门控、分析结果、图表和最终交付。
- `DB-05` 到 `DB-08` 不仅验证错误提示，还验证 Skill 是否在正确阶段停止，以及是否触发了错误的越权行为。
- `DB-09` 和 `DB-10` 不仅验证自然语言解析，还进一步验证筛选结果记录范围是否与解析结果一致。
- `DB-11` 和 `DB-12` 不仅检查最终报告文件是否存在，还检查交付质量和洞察治理这两类更容易被忽略的高风险要求。

因此，当前 12 条用例虽然高于“至少 5 条”的最低要求，但每条用例承担的覆盖职责不同，整体测试集不存在明显的重复堆砌，更符合“用更少用例覆盖更多场景”的专项目标。

## 六、当前未单列覆盖项与后续增强方向

当前测试集已经覆盖 `db_report` Skill 的核心主路径、关键失败路径、意图解析边界、红线违规、三格式交付和洞察治理要求，能够支撑本轮提交与评估使用。基于“优先用更少用例覆盖更多关键场景”的设计原则，以下内容暂未单列为独立 testcase，而被视为后续可增强方向：

- **`E2002` 不识别扩展名**：当前测试集已覆盖缺数据、文件不存在和关键指标空值，但尚未单列验证不支持扩展名时的拒绝行为。
- **`E2004` 本地数据为空或无法转成 `records`**：当前尚未单列“文件存在但内容为空、结构不合法或转换失败”的接入失败场景。
- **粘贴 JSON 的成功路径**：当前测试集已覆盖粘贴 JSON 的失败路径，但未单列“用户直接粘贴标准 records JSON 并成功生成报告”的正向用例。
- **`csv/xlsx` 的成功接入路径**：Skill 原生支持本地 `csv/xlsx`，当前主路径用例主要聚焦于 `log/json` 两类更核心、更常见的数据源。
- **`comparison` 的不可比分支**：当前已覆盖公平性核查与正常对比路径，但尚未单列“存在不可比条件时只能输出描述性分析、不得给出绝对优劣结论”的专门用例。

上述未单列项并不影响当前测试集对核心能力的覆盖完整性，但在下一轮增强中，可以优先围绕“错误码完整性”“输入类型完整性”“不可比分支治理”三类主题继续扩展，以进一步提升测试集的规范贴合度与鲁棒性。
