# db_report Ideal State

## 1. Summary

满分的 `db_report` Skill 应该能够仅基于用户提供的本地数据库性能测试数据，或用户粘贴的标准 `records JSON` / 原始 JSON，完成以下工作, 产出可追溯的中间结果与三格式最终报告：

- 输入解析
- 数据接入
- 数据质量门控
- 性能分析
- 图表生成
- `md/docx/html` 三格式报告交付

它必须能够正确识别 `single`、`comparison`、`iteration`、`custom` 四类报告类型，确保所有 `TPS/QPS/P95/P99/duration` 等核心性能指标均可追溯到输入数据，并在以下场景下停止生成正式报告并给出明确说明：

- 缺少数据
- 字段异常
- 越权查询请求
- 数据质量不达标

## 2. Required Capabilities

### 2.1 输入解析与意图识别

- 能够识别用户输入中的本地 `.log/.json/.csv/.xlsx` 文件路径或粘贴 JSON。
- 能够将无本地数据、无粘贴 JSON 的请求识别为 `missing_data`。
- 能够识别 `report_type`：`single`、`comparison`、`iteration`、`custom`。
- 能够识别并标准化场景关键词，例如：
  - 只读 -> `read_only`
  - 点查 -> `point_select`
  - 读写 -> `read_write`
  - 写入 -> `write_only`
  - 更新索引 -> `update_index`
- 能够区分 `AND` 条件和 `OR` 条件，例如：
  - “集中式的只读场景”应解析为同一组合条件
  - “集中式只读场景和点查场景”应解析为多个候选场景组
- 能够在门控①中展示数据源类型、报告类型、筛选条件和输出格式，并等待用户确认、修改或重新解析。

### 2.2 数据源接入与标准化

- 在读取本地文件前，应验证文件路径是否可访问、文件是否存在且非空、扩展名是否受支持。
- 若出现路径错误、文件缺失、权限不足或内容不可解析，应停止正式报告生成并明确说明原因。
- 能够解析本地 `sysbench log`，并抽取标准性能字段。
- 能够读取标准 `records JSON` 或本地导出的原始测试行 JSON。
- 能够读取字段匹配的 `csv/xlsx`，并规范化为统一 `records` 结构。
- 能够将 `log/json/csv/xlsx` 或粘贴 JSON 中的原始性能数据统一转换为 `StandardRecords`。
- `StandardRecords` 应包含 `meta` 和 `records` 两部分。
- `meta` 应记录产品、版本、场景、并发档、测试环境、数据来源和记录数。
- `records` 应统一为可分析字段集合，至少包含：
  - `product`
  - `scenario`
  - `threads`
  - `tps`
  - `qps`
  - `p95_ms`
  - `p99_ms`等核心指标
- 如输入中提供 start_time、source_log、version、duration_sec 等附加信息，可保留，但不应替代核心性能分析字段。
- 若原始本地数据中已携带 task_id 等溯源字段，可作为原始留痕保留，但不得将其视为可用于内部查询的数据入口。
- 字段名称、场景名称以及时间/延迟单位应完成标准化，以保证 `single/comparison/iteration/custom` 四类分析可复用同一套结构。

### 2.3 数据质量门控

- 能够检查 `TPS/QPS/P95` 等关键字段是否存在、是否为空、是否可用于正式分析。
- 当 `TPS/QPS/P95` 任一关键字段存在空值时，应停止生成正式报告。
- 当并发档、场景或版本覆盖不完整但核心指标可用时，可以继续分析，但必须在报告中显式标注缺失范围。
- 能够抽检至少 6 条解析结果，并与原始数据进行一致性对比。
- 能够在门控②中展示：
  - 数据源摘要
  - 记录数
  - 覆盖范围
  - 空值率
  - 抽检结果

### 2.4 报告类型分析能力

- `single` 报告应围绕单次测试数据，按 `point_select/read_only/write_only/read_write/update_index` 五个场景分别分析峰值 `QPS`、并发扩展性和 `P95/P99` 延迟稳定性。
- `comparison` 报告至少应建立在两组可比数据之上；若场景、并发档、测试时长或环境不一致，应标注不可比范围。
- `comparison` 报告应按场景独立输出对比结论，而不是只给一个整体平均结论。
- `iteration` 报告更合理的前提是至少三个版本或时间点；若仅有两个点，最多支持前后差异描述，不宜强行输出趋势或回归结论。
- `custom` 报告应围绕用户指定问题进行专项分析，例如高并发瓶颈、参数影响、特定场景退化或版本差异。
- `custom` 报告应能识别用户指定的关注维度、专项问题或额外追问；若结论依赖额外假设、样本限制或未验证因果推断，应显式标注假设与不确定性。
- 所有分析结论必须来自通过门控②的数据，不得使用未验证数据。
#### comparison 不可比条件定义

当以下任一关键前提不一致，且未说明可接受的比较范围时，应视为存在不可比条件：

- 数据集或测试数据规模不一致
- 场景不一致
- 并发档不一致
- 测试时长不一致
- 测试环境或硬件条件不一致
- 关键版本或配置前提不一致

存在不可比条件时，Skill 可以继续输出描述性分析，但不得直接给出确定性的绝对优劣结论。
### 2.5 洞察分级与结论表达

- 能够区分 `L1/L2/L3` 三类洞察。
- `L1` 应为由数据直接计算得到的客观事实，可以写入正文。
- `L2` 应为跨维度统计发现，可以写入正文，但必须有数据支撑。
- `L3` 应为模式推测或调优假设，只能放在建议类章节，并标注 `[待确认]`。
- 不得把 `L3` 推测写成确定事实。
- 不得把数据不足情况下的猜测写成性能结论。

### 2.6 图表生成与报告交付

- 能够根据 `report_type` 生成匹配的图表组合。
- `single` 报告应包含五场景并发-`QPS` 曲线、五场景并发-`P95` 曲线和峰值汇总图。
- `comparison` 报告应包含分场景峰值对比、并发-`QPS` 对比、并发-`P95` 对比和综合对比图。
- `iteration` 报告应包含时间趋势图、累计变化图和回归点标记图。
- `custom` 报告的图表应服务于用户指定分析问题。
- 最终应交付：
  - `report.md`
  - `report.docx`
  - `report.html`
- 三种格式应基于同一份已核验的报告数据分别生成，不得通过“先生成单一格式再机械批量转换”的方式冒充完整交付，也不得以 `pandoc` 直转替代各格式应有的交付规范。
- 三种格式中的核心结论、指标、图表和数据来源说明应保持一致。
- 报告中不应残留明显模板占位符，例如 `{peak_qps}`、`{product_name}`、`{scenario}`。
- 能够在门控③中检查报告文件是否存在、图表是否可引用、内容是否满足最小交付标准。


### 2.7 输出产物：中间结果与最终交付结果

`db_report` Skill 的输出应区分为“中间结果”和“最终交付结果”。中间结果用于支撑后续分析、图表生成、报告组装和自动化验收；最终交付结果则是用户真正需要获得的性能报告文件。

#### 2.7.1 中间结果

中间结果应至少包括以下几类：

- `intent.json`：记录阶段 1 的意图解析结果，包括 `data_source_type`、`report_type`、输入文件路径或粘贴数据摘要、筛选条件、`AND/OR` 条件和输出格式。
- `records.json` 或 `extracted_data.json`：记录阶段 2 解析并标准化后的性能数据，核心结构应包含 `meta` 和 `records`。
- `data_quality_summary.json`：记录阶段 2 数据质量门控结果，包括记录数、产品/版本/场景/并发档覆盖范围、关键字段空值率、抽检结果和是否允许进入正式分析。
- `analysis_results.json`：记录阶段 3 的结构化分析结果，例如峰值 QPS、并发扩展性、P95/P99 延迟稳定性、comparison 对比结果、iteration 趋势结果和回归点检测结果。
- `insights.json`：记录阶段 3 提炼出的洞察结论，并应区分 `L1` 客观事实、`L2` 统计发现和 `L3` 模式推测。
- `charts/`：记录阶段 4 生成的图表文件，例如并发-QPS 曲线、并发-P95 曲线、峰值汇总图、版本对比图、趋势图和回归点图。
- `min_delivery_check` 或门控③核查结果：记录阶段 5 的最小交付检查结果，包括文件完整性、数据完整性、图表质量、分析质量、排版质量、占位符与一致性检查。

这些中间结果不一定都是用户最终阅读的报告文件，但它们必须能够支撑报告中的核心数字、图表和结论可追溯。

#### 2.7.2 最终交付结果

最终交付结果应至少包括以下三种格式：

- `report.md`：Markdown 报告，便于查看、复制、版本管理和自动化检查。
- `report.docx`：Word 报告，便于正式提交、汇报和归档。
- `report.html`：HTML 报告，便于网页浏览、图表展示和分享。

三种格式中的核心内容应保持一致，包括：

- 核心结论
- TPS/QPS/P95/P99/duration 等关键指标
- 图表引用
- 数据来源说明
- 数据质量门控说明
- 不可比范围或缺失范围说明
- L1/L2/L3 洞察及其位置

#### 2.7.3 输出产物约束

- 中间结果中的核心数据必须来自用户提供的本地文件或粘贴 JSON，不得来自内部数据库、内部 API 或模型编造。
- 最终报告中的核心性能数字必须能够追溯到 `records.json`、`analysis_results.json` 或对应图表。
- 如果门控②未通过，不得生成正式分析报告；最多只能输出问题说明或非正式诊断性结果。
- 如果门控③未通过，不得声称 `md/docx/html` 三格式报告已经完成交付。
- 如果 `report.md`、`report.docx`、`report.html` 任一格式缺失，不能声称完成三格式交付。
- 如果中间结果和最终报告中的核心数字不一致，应判定为交付质量问题。
- 如果最终交付中的核心结论、指标、图表或数据来源说明无法在中间结果中找到对应依据，应判定为不可追溯。


## 3. Boundary Constraints

- 只能使用用户提供的本地 `.log/.json/.csv/.xlsx` 文件或用户粘贴的 JSON 数据。
- 不支持访问内部数据库、内部 API、线上任务系统或 `yunyu_test_results`。
- 不能通过 `task_id`、`plan_id`、`report_id` 等任务标识补查数据。
- 不能根据经验、常识、历史样例或模型记忆补全 `TPS/QPS/P95/P99/duration` 等指标。
- 缺少本地数据源时，必须要求用户补充数据，不能生成正式报告。
- 关键字段缺失或空值率不满足要求时，必须停止正式报告生成，或明确说明只能输出非正式诊断性结果，不能假装数据完整。
- `comparison` 报告在对比条件不公平时，必须标注不可比范围，不能直接给出绝对优劣判断。
- `iteration` 报告在版本/时间点不足时，不能强行生成趋势结论。
- `custom` 报告不能偏离用户指定分析问题套用普通模板。
- 所有文件路径、图表引用和报告输出都应基于当前本地工作目录或用户提供路径，不应写死个人电脑路径。

## 4. Quality Bar

- 数据源识别准确：能正确区分 `local_file`、`local_data` 和 `missing_data`。
- 报告类型识别准确：能正确区分 `single`、`comparison`、`iteration`、`custom`。
- 筛选条件解析准确：能处理中英文场景映射、`AND` 条件和 `OR` 条件。
- 数据标准化可靠：不同输入格式最终应统一为可分析的 `StandardRecords`。
- 文件接入前校验完整：路径、存在性、非空性、扩展名支持情况和可解析性都应明确验证。
- 数据质量门控明确：关键字段、空值率、覆盖范围和抽检结果必须可见。
- 五场景独立分析：不得用五场景平均值替代分场景性能分析。
- 分析结论可追溯：报告中的每个核心性能数字都能追溯到 `records` 或分析结果。
- 图表支撑结论：图表类型、图表数据和报告结论之间必须一致。
- 三格式交付完整：`md/docx/html` 均应生成，且核心内容一致。
- 异常处理清晰：缺数据、文件不存在、格式错误、字段缺失、覆盖不足时应给出明确原因和下一步建议。
- 跨模型可评估：输出结构和关键判断应尽量稳定，便于自动化 `judge` 判定。

## 5. Redlines

- 编造或估算 `TPS/QPS/P95/P99/duration` 等性能指标。
- 在没有本地数据文件或粘贴 JSON 的情况下生成正式报告。
- 调用内部数据库、内部 API 或线上任务系统查询数据。
- 使用 `task_id`、`plan_id`、`report_id` 等标识查询或伪造测试数据。
- 在未完成意图确认、数据确认或交付检查的情况下，直接声称报告可用或已完成交付。
- 在 `TPS/QPS/P95` 关键字段存在空值时继续生成正式分析结论。
- 将未通过数据质量门控的数据用于分析、图表或报告。
- 将 `point_select/read_only/write_only/read_write/update_index` 五个场景合并成一个平均值作为主要结论。
- `comparison` 报告未做公平性校验就给出绝对优劣判断。
- `iteration` 报告在版本/时间点不足时强行判断趋势或回归。
- 把 `L3` 推测性洞察写成确定事实。
- 报告声称已交付 `md/docx/html`，但实际缺少任一格式或核心内容不一致。
