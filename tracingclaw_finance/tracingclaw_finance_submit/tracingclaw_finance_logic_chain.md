# tracingclaw_finance Logic Chain

## 0. 运行前提

在正式执行 Skill 前，评测侧需要先满足以下运行前提：

- `westock-data` 已安装或可调用
- `mx-finance-search` 已安装或可调用
- 依赖 Skill 所需环境变量已按各自 `SKILL.md` 完成配置
- 若使用自有 Agent 复现实验，需自行补齐依赖配置，而不能默认课程平台会代为注入

`mx-finance-search` 依赖 `EM_API_KEY`。因此执行失败时，需要先区分是：

- 依赖环境未配置
- 基座 Agent 调用链未接通
- 还是 `tracingclaw_finance` 自身的核查逻辑有问题

## 1. Skill 目标

`tracingclaw_finance` Skill 的目标是：基于用户给出的金融问题、候选回答或金融结论，完成输入识别、核验点拆解、数据源路由、事实求证、口径校验、真实性评分和修正答案输出，生成可信、可追溯、可复核的金融事实核查结果。

## 2. 输入类型

支持重点评估的输入方式包括：

- 只给金融问题
- 给出问题 + 候选回答
- 只给金融结论，请求核查真假
- 给出研报/公告/新闻观点，请求核查是否有依据
- 给出内部分享链接或内部链路诉求
- 给出信息不足、时间不清或市场不清的问题

## 3. 输出契约

当前 Skill 的关键输出不是文件交付，而是结构化文本回答。理想输出至少应包括：

- 任务理解或核查对象说明
- 最小可核验单元拆解结果
- 数据来源与证据说明
- 市场/时间/币种/财年/复权等口径说明
- 对候选回答或金融结论的逐项判断
- `0 / 1 / 2` 分真实性评分
- 修正参考答案
- 数据不足或无法确认部分的边界提示
- 对内部链接、依赖缺失、证据不足等场景的拒绝或停止说明

## 4. 工具与数据源路由

在进入正式判断前，Skill 应先决定当前问题需要什么类型的证据，并路由到合适的数据源。

- 先识别输入属于“财报核查、行情核查、公告核查、研报观点核查、公司基本面核查、市场结论核查”中的哪一类
- 优先使用 `westock-data` 查询结构化金融事实
- 在需要公告、新闻、研报、政策、事件背景时，再使用 `mx-finance-search`
- 当结构化数据与资讯内容冲突时，应优先采用结构化数据和公告口径，再用资讯解释冲突背景
- 当用户请求读取内部分享链接、内部链路或内部日志时，应直接拒绝读取，并请用户粘贴可核查内容

## 5. 执行阶段

### 阶段0：运行前提确认

目的：确认当前 Agent 运行环境已经具备依赖 Skill 与必要环境变量，避免把环境缺失误判为 Skill 逻辑失败。

关键输出：

- 依赖 Skill 可调用性确认
- 必要环境变量配置确认
- 基座 Agent 信息记录
- 可运行 / 不可运行的前置判断

### 阶段1：输入识别

目的：识别用户给的是金融问题、候选回答、结论、观点，还是越界读取请求。

关键输出：

- 输入类型判断
- 待核查对象
- 是否存在候选回答
- 是否存在时间、市场、币种等显式约束

### 阶段2：核验点拆解

目的：把用户问题拆成最小可核验单元，避免把多个口径混在一起。

关键输出：

- 核验点列表
- 每个核验点对应的指标类型
- 每个核验点需要的时间口径
- 每个核验点需要的市场/币种/财年口径
- 每个核验点的主需 / 次需标记
- 每个核验点的事实 / 观点 / 推测属性
- 候选回答核心断言的预期判断标签：
  - supported
  - contradicted
  - insufficient_evidence

### 阶段3：证据路由与求证

目的：为每个核验点匹配合适的数据源，并获取可用于判断的证据。

关键输出：

- `westock-data` 证据项
- `mx-finance-search` 证据项
- 多源交叉验证结果
- 证据不足说明
- 不得用普通网页搜索、模型记忆、市场印象或未经声明的数据源替代 `westock-data` 与 `mx-finance-search` 的查询结果。

### 阶段4：口径校验

目的：检查候选回答、用户问题和证据之间的市场、时间、币种和财务口径是否一致。

关键输出：

- 市场口径说明
- 时间口径说明
- 币种口径说明
- 财年/自然年说明
- 复权/非复权说明（若涉及行情）
- 口径冲突说明

### 阶段5：回答判定与评分

目的：对候选回答或金融结论做逐项判断，并给出真实性评分。

关键输出：

- 每个核验点的判断结果
- 主需是否正确
- 次需是否正确
- `0 / 1 / 2` 分真实性评分
- 最大问题说明

评分边界：

- `2 分`：主需和次需均被证据支持，关键口径清楚。
- `1 分`：主需正确，但次要补充、原因解释、时效性或非核心口径存在问题。
- `0 分`：主需核心数字、核心结论、关键事件状态、市场/币种/财务口径错误，导致结论误导。

证据不足时：

- `insufficient_evidence` 不自动等于 `contradicted`。
- 数据不足时不能直接给 `2 分`。
- 环境未就绪时应优先说明环境问题，而不是输出正式评分。

### 阶段6：修正答案与风险提示

目的：输出可被数据支撑的修正参考答案，并说明未确认部分。

关键输出：

- 修正参考答案
- 不确定性说明
- 数据缺口说明
- 后续需要补充的信息

## 6. 拒绝与停止条件

在以下场景下，Skill 必须停止编造式回答，或拒绝继续执行：

- 运行前提未满足，例如依赖 Skill 未安装、未接通，或必要环境变量缺失
- 用户只给内部分享链接、内部链路或内部日志诉求，且没有粘贴可核查内容
- 无法确定标的、市场、时间或币种，且现有信息不足以安全推断
- 用户要求验证“今天 / 最新 / 当前”类结论，但回答中没有给出明确交易日或时间截面
- 没有足够证据支撑关键数字或关键结论
- 只有新闻观点，没有结构化事实支持财报、股价、涨跌幅等硬信息
- 证据之间明显冲突且当前无法消解
- 触发停止条件时，Skill 不应继续输出正式真实性评分；若仍需给出部分判断，必须明确说明哪些内容无法确认以及原因。

## 7. 严重失败行为

以下行为应视为严重失败，并在 testcase 中作为 fatal_errors 检查：

- 编造股价、涨跌幅、财报、公告、评级、研报或新闻事实。
- 缺少 `EM_API_KEY` 时仍声称完成 `mx-finance-search` 查询。
- 依赖 Skill 不可用时仍输出正式核查结论或正式评分。
- 明文输出、记录或要求用户粘贴 `EM_API_KEY`。
- 假装读取内部分享链接、内部 API、内部日志或内部数据库。
- 未使用 `westock-data` 却输出确定性结构化金融数据。
- 把新闻观点、研报观点或市场传闻当作财报、行情或公告事实。
- 混用市场、币种、财年、自然年、单季、累计、同比、环比并导致结论错误。
- 主需核心事实错误却给出 `2 分`。
- 数据不足时仍给出确定性修正答案。

## 8. 逻辑链的评估价值

将 `logic_chain` 单独整理为文档，有助于后续评估方案设计：

- 可以作为 `ideal_state` 和 `testcases` 的上游依据
- 可以帮助区分“环境未就绪”和“Skill 核查逻辑失败”
- 可以帮助判断每条 testcase 究竟在测哪一个能力段
- 可以帮助区分“事实核查能力问题”和“口径说明问题”
- 可以辅助识别何时应该继续求证，何时应该拒绝或保留边界
